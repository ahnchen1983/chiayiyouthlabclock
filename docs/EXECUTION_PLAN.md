## 5. 決策定案後的執行計畫

### Step 1（我做）：批次寫工單

決策表回來後，我同一個 message spawn 4–6 個 `Plan` subagent 平行寫工單：

- 4 份 A 批工單 → 1 個 commit + push
- 4 份 B 批工單 → 1 個 commit + push（等 A 完成後）
- 4 份 C 批工單 → 1 個 commit + push（等 B 完成後）

每批工單同一個 commit 一次寫好，你 review 看有沒有要調整。

### Step 2（你 + Codex/我）：分 3 批實作

| 批次 | 票 | 推薦執行者 | 工期 |
|------|-----|-----------|------|
| **A** | 6.3、7.5、8.2、8.3 | 我（前端管理 + 寫 code），或 4 張平分給 Codex | 2.5–3 d |
| **B** | 8.4、6.2、6.1、8.1 | 同上 | 3.5–5.5 d |
| **C** | 6.4、8.5、7.6、7.2 | 7.2 必須等所有頁面穩定再做 | 4–5 d |

**每批的驗收節奏：**

1. 4 張票 implement（可平行）
2. 一次跑 `typecheck + test + build`
3. 一次性 commit + push
4. 你跑該批的手動煙霧測試（按工單清單）
5. 通過後進下批

每批 4 次來回（implement → typecheck → manual test → confirm），3 批合計 ~12 次。加上 step 1 的 3 次（commit/review）= ~15 次完成。

## 6. subagent 使用界線

| 用途 | 適不適用 subagent | 理由 |
|------|------------------|------|
| 平行寫工單 | ✅ 強烈推薦 | 純規劃、無副作用、互不影響 |
| 平行寫獨立批的 code（如 A 批 4 張完全不重疊的票） | ⚠️ 條件性 | 必須驗證檔案完全不重疊。實際上 A 批 4 張改的檔案有部分重疊（types.ts），要小心 |
| 平行寫同批內互相依賴的票 | ❌ 不行 | 必然衝突 |
| 同時實作 + 驗收 | ❌ 不行 | 驗收必須序列化（你的瀏覽器只有一個） |

**結論**：subagent 主要砍「寫工單」階段的時間，不是「寫 code」階段。

## 7. 完成定義（DoD）

每張票要算「真的完成」必須過：

1. `npm run typecheck` 0 錯誤
2. `npm run test` 全綠（每張新功能票必須帶 ≥ 3 個新測試）
3. `npm run build` 無新警告
4. 工單裡的「手動煙霧測試」清單你勾完
5. SDD.md / CHANGELOG.md / DEVELOPMENT_ROADMAP.md 同步更新（最後一批做完一次更）

## 8. 風險與緩解

| 風險 | 緩解 |
|------|------|
| 客戶決策後改變心意 | 工單可改，但 commit 出去的 code 改成本高。每批寫工單後**先別 implement**，等你 review 過再開工 |
| Codex 跟我寫的 code 風格不一致 | 工單裡明確指定「比照既有 ScheduleManager 風格」「不要新增 Tailwind class」 |
| Sandbox 不能 push | 你每批負責 1 次 push（總共 3 次）|
| 手動煙霧測試吃時間 | 每批我把「最小必測」清單列出來，可在 15–20 分鐘內跑完 |
| Firebase 配額暴衝 | 7.6 FCM 需要新申請 Firebase Cloud Messaging API key，部署前做 |
