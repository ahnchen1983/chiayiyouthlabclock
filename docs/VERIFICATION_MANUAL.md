# 系統驗證手冊 (Verification Manual)

> **用途：** 提供可執行的測試步驟，驗證系統各項功能是否正常運作。
> **適用版本：** v1.3（Phase 1+2 完成）
> **對應文件：** SDD v1.3、DEVELOPMENT_ROADMAP.md、CHANGELOG.md

---

## 使用方式

1. 每個測試案例依序執行「操作步驟」
2. 對照「預期結果」確認功能正確
3. 若實際結果不符，記錄於「失敗說明」並回報

測試案例編號規則：`V<Phase>.<功能>.<序號>`（例：`V1.1.01` = Phase 1 第 1 項功能的第 1 個測試）

---

## 前置作業

### P.01 首次部署後建立 SuperAdmin 帳號
| 步驟 | 操作 |
|------|------|
| 1 | 開啟部署後的網站首頁 |
| 2 | 打開瀏覽器 DevTools → Console |
| 3 | 執行：`fetch('/.netlify/functions/api', { method:'POST', headers:{'Content-Type':'application/json'}, body:JSON.stringify({action:'initialize-database'}) }).then(r=>r.json()).then(console.log)` |
| 4 | 觀察 Console 輸出 |

**預期結果：**
- 首次執行：`{ body: "{\"message\":\"已建立 SuperAdmin 帳號...\"}" }`
- 再次執行：`{ body: "{\"message\":\"已初始化\"}" }`

**失敗說明：**
- 若回傳 500 → 檢查 Netlify Functions 日誌
- 若 ADMIN 帳號仍無法登入 → 檢查 Firestore `employees/ADMIN` 是否存在且 `password` 欄位包含 `:`

## Phase 1：基礎修正驗證

### V1.1 預設假人名修正（SDD #1）

#### V1.1.01 新部署不應出現假人名
| 步驟 | 操作 |
|------|------|
| 1 | 全新 Firestore 環境，執行 `initialize-database` |
| 2 | 以 `ADMIN` / `admin1234` 登入 |
| 3 | 進入「員工管理」頁面 |
| 4 | 進入「排班管理」頁面 |

**預期結果：**
- 員工列表只有一位「系統管理員（ADMIN）」
- 排班表所有格子皆為空白（無王小明、李小華等假人名）

---

### V1.2 排班逐日制（SDD #4）

#### V1.2.01 單日修改不影響同星期其他日期
| 步驟 | 操作 |
|------|------|
| 1 | 登入 SuperAdmin |
| 2 | 進入「排班管理」，切換到當月 |
| 3 | 點擊某個週三（如 4/9），修改人員為「張三」 |
| 4 | 儲存後觀察同月的其他週三（4/2、4/16、4/23、4/30） |

**預期結果：**
- 只有 4/9 顯示「張三」
- 其他週三**不受影響**（顯示模板原本的內容或空白）

**失敗說明：**
- 若所有週三都變成張三 → `update-schedule` 還在寫 `scheduleTemplate`，需檢查後端邏輯

#### V1.2.02 套用模板功能
| 步驟 | 操作 |
|------|------|
| 1 | 在排班管理頁面點擊「套用預設模板到本月」按鈕 |
| 2 | 確認提示 |

**預期結果：**
- 該月所有日期依模板填入對應星期的排班
- Firestore `dailySchedule` 該月新增 28-31 個文件

### V1.3 班別時段編輯（SDD #2）

#### V1.3.01 修改單日時段
| 步驟 | 操作 |
|------|------|
| 1 | 進入排班管理，點擊任一天 |
| 2 | Modal 內找到「上班時段」欄位，修改為 `10:00` - `20:00` |

**預期結果：**
- Modal 顯示該日時段為 `10:00-20:00`
- 總時數顯示 `10 小時`
- 月曆格子顯示時段字串

### V1.4 管理員打卡/請假（SDD #5）

#### V1.4.01 SuperAdmin/Admin 可打卡

#### V1.4.02 管理員可提交請假
| 步驟 | 操作 |
|------|------|
| 1 | 進入「我的請假」 |
| 2 | 填寫假單並送出 |

**預期結果：**
- 請假申請成功建立
- 可於「請假審核」看到自己的申請

### V2.2 密碼安全（SDD A1-A3）

#### V2.2.01 Firestore 不存明文密碼
| 步驟 | 操作 |
|------|------|
| 1 | 進入 Firestore Console |
| 2 | 檢查 `employees/<任一員工>.password` 欄位 |

**預期結果：**
- 值的格式為 `<hex>:<hex>`（salt:hash）
- **絕對不是** 明文的 `admin1234` 或 `password`

#### V2.2.02 舊密碼自動升級
| 步驟 | 操作 |
|------|------|
| 1 | 手動將某員工 password 改為明文（例如 `test1234`） |
| 2 | 以該員工登入一次 |
| 3 | 重新檢查 Firestore |

**預期結果：**
- 登入成功
- Firestore 中該欄位已自動變成 `salt:hash` 格式

#### V2.2.03 密碼強度驗證
| 步驟 | 操作 |
|------|------|
| 1 | 新增員工，初始密碼填 `1234` |

**預期結果：**
- 前端 alert：「至少 8 字元，需含英文與數字」
- 無法建立

#### V2.2.04 登入鎖定
| 步驟 | 操作 |
|------|------|
| 1 | 對某帳號連續輸入錯誤密碼 5 次 |
| 2 | 第 6 次使用**正確**密碼 |

**預期結果：**
- 第 6 次登入被拒，錯誤訊息「帳號已鎖定，請 XX 分鐘後再試」
- Firestore `loginAttempts/<empId>` 有 failCount=5 紀錄
- 15 分鐘後可重新登入

## 前後比對測試（v1.0 → v1.3）

| 項目 | v1.0 行為（有問題） | v1.3 行為（已修正） |
|------|--------------------|---------------------|
| 初始化 | 自動寫入 5 位假員工+假人名排班 | 只建立 ADMIN + 空白模板 |
| 排班修改 | 改單日會連動所有同星期 | 只影響該日 |
| 班別時段 | 前端不可編輯 | Modal 可編輯起訖時間 |
| 管理員打卡 | 無此功能 | 側邊欄「我的打卡」 |
| IP 驗證 | 寫死 127.0.0.1 | 後端取真實 IP |
| 密碼儲存 | Firestore 明文 | scrypt 雜湊 |
| 密碼強度 | 4 字元即可 | 8 字元 + 英數 |
| 登入失敗 | 無限制 | 5 次鎖 15 分鐘 |
| 重設密碼 | alert 顯示明文 | 不顯示，只通知員工 |
| 薪資權限 | 所有 Admin 可看 | 僅 SuperAdmin |
| 操作紀錄 | 無 | auditLogs 全記錄 |
| 休館值班 | 無法安排 | 休館(值班) 狀態 |
